# LightRPC 架构设计

## 概述

LightRPC 是一个基于 C++17 的轻量级 RPC 框架，采用经典的 Reactor 网络模型，结合 Protobuf 序列化和线程池，提供高效、可靠的远程过程调用能力。

## 核心组件

### 1. EventLoop

**职责**：事件循环核心，管理所有 IO 事件和定时器。

**设计要点**：
- 单线程模型，所有事件回调在同一个线程执行
- 使用 `eventfd` 实现跨线程任务唤醒
- 内置 `TimeWheel` 定时器，支持毫秒级定时任务
- 线程安全的任务队列 `pending_functors_`

**关键方法**：
- `Loop()`：启动事件循环，阻塞直到 `Quit()` 被调用
- `Quit()`：退出事件循环
- `RunInLoop()`：在事件循环线程中执行任务（如果已在循环线程则直接执行）
- `QueueInLoop()`：将任务加入队列，由事件循环线程执行
- `AddTimer()` / `RemoveTimer()`：定时器管理

### 2. Channel

**职责**：封装单个文件描述符的事件处理。

**设计要点**：
- 每个 Channel 绑定一个 fd 和一个 EventLoop
- 维护当前关注的事件类型（EPOLLIN / EPOLLOUT）
- 提供回调机制：read_callback_, write_callback_, close_callback_, error_callback_
- 通过 EventLoop 更新 epoll 事件

**关键方法**：
- `EnableReading()` / `EnableWriting()`：启用读/写事件
- `DisableReading()` / `DisableWriting()`：禁用读/写事件
- `HandleEvent()`：处理 epoll 返回的事件

### 3. Epoller

**职责**：epoll 封装，管理多个文件描述符的事件监听。

**设计要点**：
- 使用 `epoll_create1()` 创建 epoll fd
- `AddFd()` / `ModFd()` / `DelFd()` 管理监听的 fd
- `Wait()` 阻塞等待事件，返回触发的事件数量
- 静态方法 `SetNonBlocking()` 设置 fd 为非阻塞模式

### 4. TcpConnection

**职责**：封装单个 TCP 连接。

**设计要点**：
- 使用 `shared_ptr` 管理生命周期，防止在回调中被销毁
- 维护连接状态：Disconnected / Connecting / Connected / Disconnecting
- 读写缓冲区使用 `Buffer` 类
- 连接断开时通过回调通知上层

### 5. RpcServer

**职责**：RPC 服务端，管理服务注册和请求处理。

**设计要点**：
- 内部创建 `EventLoop`、`TcpServer`、`ThreadPool`
- `Start()` 启动事件循环线程和 TCP 监听
- `Stop()` 优雅停止，等待线程退出
- `Register<ReqT, RspT>()` 强类型服务注册，自动处理序列化/反序列化
- 请求在 `ThreadPool` 中并行处理

**处理流程**：
1. `TcpServer` 接受新连接 → 触发 `onConnection`
2. 连接收到数据 → 触发 `onMessage`
3. `Codec::decode` 解析帧 → 提取 `RpcRequest`
4. 线程池执行 `RpcService::Invoke` → 调用业务 handler
5. `Codec::encode` 编码响应 → 发送回客户端

### 6. RpcClient

**职责**：RPC 客户端，管理连接和请求发送。

**设计要点**：
- 内部创建 `EventLoop` 和 `TcpClient`
- `Start()` 启动事件循环线程并自动连接
- `Stop()` 断开连接并清理所有待处理请求
- `Call<ReqT, RspT>()` 强类型调用，自动处理序列化/反序列化
- 请求超时机制，超时后自动清理 `pending_calls_`

**处理流程**：
1. `Start()` → 启动事件循环线程 → 调用 `TcpClient::Connect()`
2. `Call()` → 序列化请求 → 生成 request_id → 加入 `pending_calls_` → 发送
3. 收到响应 → `Codec::decode` → 根据 request_id 查找 callback → 执行回调
4. 超时/断开 → `DropAllPending` → 清理所有 pending calls

### 7. RpcService

**职责**：服务方法注册表。

**设计要点**：
- 线程安全的方法注册和查找
- 使用 `service_name.method_name` 作为 key
- `Invoke()` 根据请求查找并执行对应的 handler

### 8. Codec

**职责**：协议编解码。

**设计要点**：
- 帧格式：4 字节长度（网络字节序）+ Protobuf 数据
- `encode()`：将消息序列化为带长度前缀的字节流
- `decode()`：从字节流中提取完整的消息

## 协议格式

### 帧结构

```
┌─────────────────────────────────────────────────┐
│  4 字节长度 (Big-Endian)  │  N 字节 Protobuf 数据  │
└─────────────────────────────────────────────────┘
```

### RpcRequest

```protobuf
message RpcRequest {
  string service_name = 1;   // 服务名称
  string method_name  = 2;   // 方法名称
  string request_id   = 3;   // 请求 ID（唯一标识）
  bytes body          = 4;   // 业务请求体（序列化后的 Protobuf）
}
```

### RpcResponse

```protobuf
message RpcResponse {
  string request_id = 1;     // 请求 ID（与请求对应）
  int32  code       = 2;     // 错误码（0=成功）
  string msg        = 3;     // 错误信息
  bytes  body       = 4;     // 业务响应体（序列化后的 Protobuf）
}
```

## 设计原则

1. **单线程 EventLoop**：所有 IO 事件在同一个线程处理，避免锁竞争
2. **线程安全的任务传递**：通过 `eventfd` 和任务队列实现跨线程通信
3. **类型安全**：使用 SFINAE 在编译期校验 Protobuf 类型
4. **优雅退出**：`Stop()` 等待所有线程退出，确保资源正确释放
5. **零拷贝设计**：`Buffer` 使用指针操作，避免不必要的内存拷贝
